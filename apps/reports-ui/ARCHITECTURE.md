# Architecture

## Goal

Provide a lightweight, public-facing reporting UI for outputs generated by `go-no-go-engine`, plus a thin ad-hoc evaluation surface that delegates evaluation back to the engine.

## Principles

- Read-only first.
- Deterministic rendering from generated artifacts.
- No duplicated decision logic.
- No overengineering.

## System Context

- Producer: `go-no-go-engine` (CLI) generates report artifacts and executes ad-hoc evaluation requests.
- Consumer: `go-no-go-reports-ui` reads those artifacts, renders views, and can request a one-off evaluation by delegating to the engine.
- UI still does not implement decision logic locally.

## Runtime Model

Jaspr server-mode app with filesystem-backed report loading and a thin server-side bridge to the engine CLI for ad-hoc evaluation.

High-level flow:

1. Engine run writes artifacts into a reports root (engine default in this monorepo: `services/engine/output/`).
2. UI reads available artifacts from configured root.
3. UI maps artifacts into view models.
4. UI renders report pages (runs, batch, weekly, context, trends).
5. Optional ad-hoc evaluation submits URL/raw text to the server.
6. The server runs `services/engine` `evaluate-input` with JSON output and a YAML analysis artifact for one persona or all configured personas.
7. UI renders the returned evaluation summaries and keeps the artifacts under the reports root.
8. The evaluate page also builds a reusable URL history from processed job YAML and ad-hoc evaluation YAML artifacts.

## Directory/Data Assumptions

Expected report root structure (evolves with engine):

- `batch-evaluation-*.json`
- `batch-evaluation-*.md`
- `weekly*.md`
- `trend-history-*.yaml`
- `trend-alerts-*.json`
- `company-context/*.txt`
- `ad-hoc-evaluations/*.yaml`
- `jobs/**/*.yaml`

Target contract improvement:

- `runs/index.json` as canonical run registry.

## Modules (MVP plan)

- `ReportSource`: reads files from configured root.
- `ReportIndex`: discovers available runs/report artifacts.
- `ReportMapper`: maps raw artifacts to UI view models.
- `Views`: Runs, Batch, Job Detail, Weekly, Company Context, Trends/Alerts, Evaluate.
- `EngineCatalogRepository`: loads persona/candidate-profile selector options from engine config.
- `EngineEvaluationRunner`: delegates ad-hoc evaluation requests to the engine CLI.
- `EvaluationUrlHistoryRepository`: builds reusable URL history from processed job and ad-hoc evaluation artifacts.

## Config

Planned runtime config:

- `REPORTS_ROOT` (path to engine output)
- `ENGINE_ROOT` (path to the engine project, defaults to sibling `services/engine`)
- `ENGINE_GRADLEW` (override Gradle wrapper command)
- Optional `PUBLIC_BASE_URL` for deployed links

Current local default behavior:

- prefer `../../services/engine/output` when running from `apps/reports-ui` inside this monorepo
- otherwise fall back to local `output/`

## Non-Goals (Current Phase)

- No auth layer.
- No write/edit workflows.
- No direct decision scoring in UI.
- No scheduler orchestration from UI.

## Security/Politeness Notes

- UI itself does not crawl external websites.
- Crawling politeness remains responsibility of engine fetch layer.
- Avoid exposing sensitive local paths in rendered errors/logs.
