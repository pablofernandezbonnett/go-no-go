# Architecture

## Goal

Provide a lightweight, public-facing reporting UI for outputs generated by `go-no-go-engine`.

## Principles

- Read-only first.
- Deterministic rendering from generated artifacts.
- No duplicated decision logic.
- No overengineering.

## System Context

- Producer: `go-no-go-engine` (CLI) generates report artifacts.
- Consumer: `go-no-go-reports-ui` reads those artifacts and renders views.
- Future: optional CLI trigger integration from UI (out of current scope).

## Runtime Model

Jaspr server-mode app with filesystem-backed report loading.

High-level flow:

1. Engine run writes artifacts into a reports root (default: `output/`).
2. UI reads available artifacts from configured root.
3. UI maps artifacts into view models.
4. UI renders report pages (runs, batch, weekly, context, trends).

## Directory/Data Assumptions

Expected report root structure (evolves with engine):

- `batch-evaluation-*.json`
- `batch-evaluation-*.md`
- `weekly*.md`
- `trend-history-*.yaml`
- `trend-alerts-*.json`
- `company-context/*.txt`

Target contract improvement:

- `runs/index.json` as canonical run registry.

## Modules (MVP plan)

- `ReportSource`: reads files from configured root.
- `ReportIndex`: discovers available runs/report artifacts.
- `ReportMapper`: maps raw artifacts to UI view models.
- `Views`: Runs, Batch, Job Detail, Weekly, Company Context, Trends/Alerts.

## Config

Planned runtime config:

- `REPORTS_ROOT` (path to engine output)
- Optional `PUBLIC_BASE_URL` for deployed links

## Non-Goals (Current Phase)

- No auth layer.
- No write/edit workflows.
- No direct decision scoring in UI.
- No scheduler orchestration from UI.

## Security/Politeness Notes

- UI itself does not crawl external websites.
- Crawling politeness remains responsibility of engine fetch layer.
- Avoid exposing sensitive local paths in rendered errors/logs.
